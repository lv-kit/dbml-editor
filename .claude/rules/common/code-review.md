# code-review（レビュー観点の正本）

> **正本**: レビュー観点とこのプロジェクトでやりがちな問題。PR テンプレの観点も参照。

## レビュー観点（優先順）

1. **セキュリティ・データ整合性・認可**（最優先。SQLインジェクション/XSS、認可漏れ）
2. **要件との一致**: スコープ外変更が混ざっていないか（「ついでに直す」禁止）
3. **正しさ**: 境界値・失敗系・並行性。外部入力の検証
4. **設計**: 1モジュール1責務、副作用の局所化、命名（役割を表す完全な英単語）
5. **テスト**: 挙動変更に対応するテスト、バグ修正の再発テスト、拒否/失敗系
6. **可読性・保守性**: コードを言い換えるだけのコメント・TODO・未使用コードが残っていないか

## このプロジェクトでやりがちな問題

- サーバ専用処理（DB・秘密情報・認可）がクライアント到達コードに漏れている
- `web/` と `dbml-studio/` で共有すべきロジックを片方だけ変更し仕様差分を作る
- `any` / 無根拠な型アサーション / `@ts-ignore` の混入
- Svelte 変更後に `svelte-autofixer` を通していない
- shadcn/既存トークンを使わず独自スタイルを増やす
- `console.log` / デバッグ出力の残置（hook が警告）

## 判定

- **PASS** / **要確認**（指摘はあるがブロッカーでない） / **修正必要**（マージ前に対応必須）
- 判定には必ず「次アクション（推奨コマンド/エージェント）」を添える

## 完了ゲート

- 実装後、完了報告・引き継ぎ・コミット・PR の前に、必ず `code-reviewer` を実差分へ実行する
- 変更内容に応じて `security-reviewer`、`svelte-reviewer`、`database-reviewer` などの専門 reviewer も実行する
- 指摘が1件でもあれば完了にしない。指摘を修正し、影響するテスト・検査を再実行したうえで、同じ reviewer を再実行する
- すべての reviewer が指摘ゼロになるまでこのループを続け、未検証領域は検査コマンドまたは根拠を引き継ぎノートへ記録する
