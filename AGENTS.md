# 実装ルール

## 優先順位

1. セキュリティ、データ整合性、認可
2. 要件を満たす最小の変更
3. AI と人が追跡しやすい構造
4. 可読性、保守性、テスト容易性

要件に曖昧さや不足があり、実装結果を大きく左右する場合は、実装前に不足点を示して確認する。既存の挙動、公開 API、データ形式は、明示的な要件がない限り変更しない。

## プロジェクト境界

- `web/` は SvelteKit の Web アプリ、`dbml-studio/` は Tauri のデスクトップアプリとして扱う。
- ブラウザ固有、Tauri 固有、サーバー固有の API を共有モジュールに持ち込まない。
- サーバー専用の処理は `web/src/lib/server/` または `+page.server.ts`、`+server.ts` に置く。クライアントから認可、秘密情報、DB へ直接アクセスしない。
- 複数箇所で再利用する純粋なドメインロジックは、依存の少ない `src/lib/` モジュールへ抽出する。アプリ間で同一ロジックを変更する場合は、片方だけを修正して仕様差分を作らない。

## TypeScript と設計

- TypeScript を使い、`any`、無根拠な型アサーション、`@ts-ignore` は使わない。外部入力は `unknown` として検証してから型を絞り込む。
- 名前は役割を表す完全な英単語にする。略語、文脈依存の `data`、`utils`、`helper`、`handle` のような曖昧な名前を公開 API に使わない。
- 1 モジュールは 1 つの責務にする。ファイルが大きくなり、独立して説明できる責務がある場合は分割する。
- 関数は入力、出力、副作用を明確にする。副作用は境界に寄せ、パース、検証、権限判定、変換は可能な限り純粋関数にする。
- コメントは「なぜ」を補足するときだけ記述する。コードを言い換えるコメント、放置された TODO、未使用コードを残さない。
- deprecated API は使わない。依存ライブラリやフレームワークの API を新規利用する前に、現在の公式ドキュメントを確認する。

## Svelte と UI

- Svelte コンポーネントは表示、入力、状態、外部連携を混在させすぎない。独立してテストできるロジックは `.ts` へ分離する。
- UI は既存の shadcn コンポーネントを優先し、Tailwind CSS を使う。色は既存トークンを再利用し、コントラストを確保する。
- イベントから取得する値とフォーム値は、境界で型と内容を検証する。
- Svelte コードを変更したら、Svelte MCP の `svelte-autofixer` を問題がなくなるまで実行する。

## テストと検証

- 挙動を変更する実装は、先に失敗するテストを追加または更新してから実装する。
- バグ修正には再発テストを付ける。認可、入力検証、データ更新には正常系と拒否・失敗系を含める。
- パッケージマネージャーは pnpm に統一する。`npm`、`bun`、`yarn` のコマンド、ロックファイル、設定を新たに追加しない。
- コード変更後は対象テスト、`pnpm run check`、`pnpm run lint`、全テストを実行する。
- `web/` を変更した場合は `web/` で `docker compose up --build` を実行し、コンテナ内で検証する。UI を変更した場合は、日本語フォントを確認したうえで Playwright のスクリーンショットを撮影する。

## 変更の完了条件

- 要件に関係しない整形や依存更新を混ぜない。
- 変更したコード、テスト、設定の関係を追跡できる状態にする。
- 変更後に lint、型検査、テストがすべて成功していることを確認する。

---

## Svelte MCP

Svelte または SvelteKit に関する作業では、最初に `list-sections` を実行し、関連する全セクションを `get-documentation` で取得する。Svelte コードを書いた場合は `svelte-autofixer` を問題がなくなるまで実行する。プロジェクト内のファイルを書いた場合は Playground link を生成しない。
